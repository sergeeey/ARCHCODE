# ARCHCODE v2.0 — Исследование стратегических направлений для ревизии препринта

## Контекст проекта (актуальный на 2026-02-28)

ARCHCODE — аналитический mean-field симулятор экструзии петель хроматина для выявления структурно патогенных вариантов. **Препринт v1.0 загружен на bioRxiv, DOI pending.**

### Текущее состояние

- **Данные**: 1,103 реальных варианта HBB из ClinVar (353 Pathogenic/LP + 750 Benign/LB)
- **Физическое ядро**: Аналитическое mean-field, Kramer kinetics (α=0.92, γ=0.80, k_base=0.002)
- **Результаты**: 20 pearl-вариантов (VEP-слепых), ROC AUC=1.000, SSIM стратификация по категориям
- **Валидация**: Hi-C корреляция r=0.16 (p=0.30, n=12) — не значимо
- **Локус**: HBB (chr11:5,210,000–5,240,000, 30 kb, 600 bp resolution, 50 bins)
- **Клеточная линия**: GM12878 ChIP-seq (лимфобластоидная, НЕ эритроидная)
- **Код**: TypeScript (engine) + Python (validation), открытый: github.com/sergeeey/ARCHCODE

### Критики v1.0 (5-экспертная оценка, средняя 6.4/10)

| #   | Критика                                                                                                                                  | Серьёзность | Статус                        |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------- | ----------- | ----------------------------- |
| 1   | **Circular AUC**: категория варианта = вход модели (occupancy scaling) И ground truth (ClinVar). AUC=1.000 — тавтология, не предсказание | КРИТИЧЕСКАЯ | Caveat в тексте, но не решено |
| 2   | **GM12878 ≠ erythroid**: HBB экспрессируется в эритроидных клетках, а ChIP-seq взят из лимфобластоидных                                  | ВЫСОКАЯ     | Указано как limitation        |
| 3   | **Ручная калибровка**: α=0.92, γ=0.80 без фиттинга к данным                                                                              | СРЕДНЯЯ     | Указано как limitation        |
| 4   | **Нет клинической полезности**: инструмент для гипотез, не для диагностики                                                               | ОЖИДАЕМАЯ   | Корректно заявлено            |
| 5   | **Упрощённая физика**: mean-field, нет A/B компартментов, YY1/LDB1 петель                                                                | СРЕДНЯЯ     | Архитектурное ограничение     |

### Принципы v2.0 (из ROADMAP_v2.md)

- **"Fit on A, Predict on B"** — параметры фитятся на HBB, валидируются на SOX2 (другая хромосома)
- **Pre-registration via git** — код коммитится ДО запуска валидации
- **Honest reporting** — если SOX2 r < 0.15, сообщаем честно
- **Timeline**: 4-6 недель на всё

## Направление 1: Within-Category ROC — ответ на circular AUC

**Почему это ПРИОРИТЕТ №1:** Circular AUC — единственная критика, которая может потопить paper при рецензировании. Рецензент скажет: "ваша модель присваивает -80% occupancy nonsense-вариантам и -2% intronic — конечно AUC=1.0, это заложено в дизайн." Нужно показать, что ПОЗИЦИЯ варианта внутри одной категории влияет на SSIM.

**Исследуемые вопросы:**

- Среди intronic вариантов (9 pathogenic vs ~658 benign): различается ли распределение SSIM?
- Среди missense вариантов: зависит ли SSIM от расстояния до ближайшего CTCF-сайта?
- Можно ли построить within-category ROC с AUC > 0.5 (т.е. лучше случайного)?
- Какой минимальный размер выборки нужен для статистически значимого within-category AUC?
- Если within-category AUC ≈ 0.5 — что это значит для модели и как честно это раскрыть?

**Методология:**

- Разбить Combined Atlas (1,103 варианта) на подгруппы по категории
- Для каждой подгруппы с >5 вариантами в обоих классах — ROC с bootstrap CI
- Показать зависимость SSIM от геномной позиции (scatter plot: position vs SSIM, coloured by label)
- Permutation test: перемешать labels внутри категории, пересчитать AUC 10,000 раз

**Ожидаемый результат:** Честная оценка — есть ли у ARCHCODE позиционный сигнал сверх категориального?

## Направление 2: Erythroid-Specific Data (K562 Micro-C + CUT&Tag)

**Почему это ПРИОРИТЕТ №2:** Использование GM12878 для HBB-локуса — биологически некорректно. HBB активно экспрессируется в эритроидных клетках, а GM12878 — лимфобластоидная линия с другим хроматиновым ландшафтом. K562 — эритролейкемическая линия с активной экспрессией HBB.

**Исследуемые вопросы:**

- Где найти открытые матрицы Micro-C или Hi-C для K562 в регионе HBB? (4D Nucleome, ENCODE 4, GEO)
- Какое разрешение доступно? (500 bp, 1 kb, 5 kb)
- Есть ли K562 CUT&Tag или ChIP-seq для CTCF и MED1?
- Как изменится корреляция r если заменить GM12878 на K562?
- Micro-C vs conventional Hi-C: какой метод лучше для мелких петель (30 kb окно)?

**Источники для поиска:**

- 4D Nucleome Data Portal: https://data.4dnucleome.org/ (фильтр: K562, Micro-C)
- ENCODE 4: https://www.encodeproject.org/ (фильтр: K562, ChIP-seq, CTCF/MED1)
- GEO: запрос "Micro-C K562" или "Hi-C K562 HBB"
- Krietenstein et al. 2020 (Micro-C methodology)

**Применимость:** Замена occupancy profiles в `config/default.json` — CTCF positions и enhancer regions из K562 вместо GM12878.

## Направление 3: Cross-Locus Bayesian Validation (HBB → SOX2)

**Почему это ПРИОРИТЕТ №3:** Ручная калибровка α, γ — методологическая слабость. Bayesian optimization на HBB с валидацией на SOX2 превращает "ручную калибровку" в "cross-validated parameter estimation."

**Исследуемые вопросы:**

- Gaussian Process (scikit-optimize `gp_minimize`) vs Tree-Parzen (Optuna): что лучше для 3 параметров?
- Objective function: L2 distance vs Pearson correlation vs SSIM между моделью и Micro-C?
- Сколько итераций нужно для сходимости при 3 параметрах? (α, γ, k_base)
- Какой диапазон поиска разумен? (α: [0.5, 1.5], γ: [0.5, 1.2], k_base: [0.0005, 0.01])
- Критерий успеха: HBB training r > 0.5, SOX2 validation r > 0.3 — реалистично ли это?

**Зависимости:** Требует Направление 2 (Micro-C данные для обоих локусов).

**Anti-overfitting:**

- Параметры фитятся ОДИН РАЗ на HBB
- SOX2 — другая хромосома (chr3), другой ген, другая архитектура петель
- Код коммитится ДО запуска валидации (pre-registration via git timestamp)

## Направление 4: TDA (Persistent Homology) как дополнение к SSIM

**Почему это полезно, но НЕ приоритет:** SSIM — пиксельная метрика. Сдвиг пика на 1 бин (600 bp) может радикально изменить SSIM, хотя топология петли сохранена. Wasserstein distance на persistence diagrams решает эту проблему, но это техническое улучшение, а не ответ на главные критики.

**Исследуемые вопросы:**

- Ripser vs Gudhi vs Giotto-TDA: какая библиотека лучше для Python пайплайна?
- H0 (connected components) vs H1 (loops): какой Betti number информативнее для петель хроматина?
- Коррелирует ли Wasserstein distance с SSIM? Если r > 0.9 — нет смысла добавлять
- Есть ли случаи где TDA ловит сигнал, а SSIM нет (и наоборот)?
- Вычислительная сложность: сколько секунд на матрицу 50×50?

**Научная база для поиска:**

- Cang & Wei, 2020 — TDA для белковых структур
- Статьи о persistent homology в 3D-геномике (если существуют)
- Обзоры TDA в биоинформатике

**Применимость:** Добавить `topology_distance` как колонку рядом с `ARCHCODE_SSIM` в атласе.

## Направление 5: Масштабирование на 2-3 дополнительных локуса

**Почему НЕ genome-wide:** 50,000 VUS genome-wide — это v3.0+. Для v2.0 достаточно 2-3 локусов с хорошо документированной 3D-архитектурой, чтобы показать generalizability.

**Кандидаты для масштабирования:**

| Локус      | Хромосома | Болезнь                     | Механизм                | Почему подходит                     |
| ---------- | --------- | --------------------------- | ----------------------- | ----------------------------------- |
| SOX2/EPHA4 | chr3      | Пороки развития конечностей | TAD boundary disruption | Lupiáñez et al. 2015, хорошо изучен |
| MYC        | chr8      | Лимфомы, CRC                | Enhancer hijacking      | Super-enhancer + CTCF anchors       |
| CFTR       | chr7      | Муковисцидоз                | Splice + regulatory     | Хорошо аннотирован, много VUS       |

**Исследуемые вопросы:**

- Какие CTCF-сайты и энхансеры аннотированы для каждого локуса в ENCODE?
- Есть ли Micro-C/Hi-C данные для этих регионов в K562 или релевантных клетках?
- Как адаптировать 30 kb окно ARCHCODE под разные размеры TAD? (CFTR TAD ~200 kb)
- Сколько ClinVar вариантов доступно для каждого локуса?

**Реалистичный план:** SOX2 уже запланирован как validation locus (ROADMAP Phase 2). MYC или CFTR — как третий, если SOX2 валидация успешна.
